# WebBrowser 技术方案

一款基于 **WPF + WebView2 + WPF-UI** 的个人轻量浏览器壳子。刻意精简功能（无广告拦截、无扩展），把工程精力集中在三个品质维度：**极致 UI 响应、内存控制、Windows 原生融合感（Mica 毛玻璃 + 流畅拖拽）**。

---

## 3. 核心架构决策

### 3.1 唯一共享 `CoreWebView2Environment`

[Services/WebViewEnvironmentService.cs](Services/WebViewEnvironmentService.cs) 是单例，懒创建并缓存**唯一**环境，`UserDataFolder = %LOCALAPPDATA%\WebBrowser\WebView2`。所有标签 `EnsureCoreWebView2Async(env)` 复用它，共享 browser / GPU / network 进程（renderer 按站点隔离决定是否共享）。

- 用 `SemaphoreSlim(1,1)` 串行化首次创建，防止并发首开重复 `CreateAsync`（多环境指向同一 UserDataFolder 会进程抖动，WebView2Feedback #3378）。
- 订阅 `BrowserProcessExited`：该事件是**全局**的，触发时所有标签的 Core 都已死，需走重建路径而非逐标签恢复。
- 暴露 `ProcessInfosChanged` 供遥测观察进程增减。

### 3.2 WebView2 实例归属：`WebViewTab` 持有，视图"领养"而非"创建"

**绝对不要**把 `WebView2` 放进 stock `TabControl` 的 `ContentTemplate`——它在切换时会回收/重置内容宿主，把 WebView2 重新 unparent，这是响应迟钝和进程泄漏的元凶。

做法：`WebViewTab`（非 VM、非 View）在 UI 线程一次性 `new WebView2()`；VM 持其引用；内容区用 **overlay `ItemsControl`**，所有标签的 WebView2 同处一个 `Grid`、**全生命周期保持 parented**，仅靠 `IsActive` 切换 `Visibility`：

```xml
<ItemsControl ItemsSource="{Binding Tabs}">
  <ItemsControl.ItemsPanel><ItemsPanelTemplate><Grid/></ItemsPanelTemplate></ItemsControl.ItemsPanel>
  <ItemsControl.ItemTemplate>
    <DataTemplate>
      <ContentPresenter Content="{Binding WebView.Control}"
                        Visibility="{Binding IsActive, Converter=...}"/>
    </DataTemplate>
  </ItemsControl.ItemTemplate>
</ItemsControl>
```

切标签永不重建控件、永不 reparent HWND。

### 3.3 标签生命周期与拆卸时序（内存释放的核心）

[Services/TabLifecycleService.cs](Services/TabLifecycleService.cs) 拥有 `ObservableCollection<TabViewModel>` 与活动标签，负责：

- **开标签**：建 VM → 加入集合 → SetActive → `GetEnvironmentAsync()` → `WebViewTab.InitializeAsync`（EnsureCore + 配置 Settings + HookEvents）→ 导航。
- **切标签**：eager Resume 新活动标签；旧标签 debounce 12s 后 TrySuspend（见 3.5）。
- **关标签**（顺序敏感，每步 try/catch）：
  1. 若关的是活动标签，先选邻居（右无则左）激活；若是最后一个标签，新开一个空白标签（Chrome/Edge 行为）。
  2. `Tabs.Remove(tab)` → ContentPresenter 自动 unparent 控件；`CancelSuspendTimer(tab)`。
  3. `await tab.WebView.TearDownAsync()`：退订全部事件 → 若已挂起先 Resume → `Control.Dispose()`（WPF WebView2 是 HwndHost，Dispose 销毁宿主窗口 → 关闭 CoreWebView2Controller → 终止该标签 renderer 进程）。
  4. `GC.Collect` + `WaitForPendingFinalizers` 促使渲染进程尽快退出。
- **关应用**：`OnClosing` 是同步的，用 **cancel-then-await 模式**（关键修复）：

```csharp
protected override async void OnClosing(CancelEventArgs e)
{
    if (_isClosing) { base.OnClosing(e); return; }
    _isClosing = true;
    e.Cancel = true;
    try { if (DataContext is MainViewModel vm) await vm.ShutdownAsync(); } catch { }
    await Task.Yield();   // 让 WPF 完成"中止首次关闭"后再发起第二次 Close，否则抛 "window is closing"
    Close();
}
```

### 3.4 下载管理器：接管 + 状态机

[Services/DownloadManagerService.cs](Services/DownloadManagerService.cs) 集中处理每个标签 `Core.DownloadStarting`（一处接线），`e.Handled = true` 抑制默认下载条。处理流程：

- 文件名取自 `e.ResultFilePath`（失败回退到 `op.Uri` 末段）。
- **真实 Downloads 目录**用 `SHGetKnownFolderPath(FOLDERID_Downloads)` 解析（**不要**用 `%USERPROFILE%\Downloads` 启发式——用户可能重定位，例如本机真实路径是 `D:\Downloads`），并 `Directory.CreateDirectory`。
- `ResolveUniquePath` 处理重名：`file (1).ext`、`file (2).ext`…
- `e.ResultFilePath` 只能在 handler 返回前设一次（#2890）。
- 建 `DownloadItemViewModel` → 订阅 `op.BytesReceivedChanged` / `op.StateChanged`（**均在工作线程触发**，必须 `Dispatcher.BeginInvoke` 再动绑定属性）→ 插入集合（最新在前）。

[ViewModels/DownloadItemViewModel.cs](ViewModels/DownloadItemViewModel.cs) 状态机：`NotStarted → InProgress →（Paused）→ Completed | Cancelled | Interrupted`。
- `Paused` / `Cancelled` 是 **UI 层本地状态**（`CoreWebView2DownloadState` 枚举只有 `InProgress/Interrupted/Completed` 三态，暂停/取消不改变底层 State）。
- 命令：Pause / Resume / Cancel / Open / Reveal（资源管理器定位）/ Retry（重新在活动标签打开源 URL）。
- **byte 计数类型坑**：`DownloadOperation.BytesReceived` 在 WinRT 投影与 COM 包装之间类型不一（`long`/`ulong`/`ulong?`，WPF 临时工程还会误报）。用 `ToInt64(object?)`（boxing + `IConvertible`）稳健转换，null（未知总大小）→ 0 → 进度条不定式。

### 3.5 内存：挂起/恢复 + 遥测

- **关标签 = 立竿见影**：每关一个标签，其 renderer 进程直接退出（M2 验证：每标签 −1 进程，连开连关零泄漏）。
- **后台标签挂起 = 省内存前提**：`TabLifecycleService` 切走标签后 debounce 12s 调 `WebViewTab.TrySuspendAsync`（`Core.TrySuspendAsync`），切回时 eager `Core.Resume`。以 `Core.IsSuspended` 为状态源。
  - TrySuspendAsync 冻结渲染进程（停定时器/脚本），让 **OS 在内存压力下**能换出挂起进程；**即时**工作集削减仅几个百分点（轻载场景在噪声内），这是其固有性质——配合 OS 压力回收才有大头。
- **遥测**：`MainViewModel` 用 `DispatcherTimer`（3s）汇总 `WebViewEnvironmentService.ProcessInfos` 各进程 `WorkingSet64` → `WebViewMemoryMb`，工具栏实时显示 `wv2: N · M MB`，便于在真实负载下观察。

### 3.6 视觉打磨（M4）

- **favicon**：`WebViewTab` 订阅 `Core.FaviconChanged` → `Core.GetFaviconAsync(Png)` → `BitmapImage`（`CacheOption.OnLoad` 脱离流 + `Freeze` 跨线程）→ 按主机名缓存，设到 `TabViewModel.Favicon`。标签头 `Image` 在 `Favicon` 为 null 时自动 `Collapsed`（不留空白）。
- **系统主题联动**：`App.OnStartup` 读注册表 `AppsUseLightTheme` → `ApplicationThemeManager.Apply(主题, WindowBackdropType.Mica, true)`。工具栏切换按钮用**两个叠加 ui:Button**（月=暗、日=亮），inline 设 `Icon`、用 `Visibility` 的 `DataTrigger` 互斥显示。
  - ⚠️ WPF-UI 已知怪癖：`ui:Button.Icon` 用 **Style setter** 设值不生效，必须 inline 设。
- **标签条并入标题栏**：`<views:TabStripView>` 放进 `<ui:TitleBar.Header>`（标签左、字幕按钮右自动渲染、中间空白可拖拽），收掉独立行，与 Mica 一体。
- 关键交互元素加 `AutomationProperties.Name`（无障碍 + 可程序化定位）：标签按钮名 = 页面标题，地址栏 / 新建 / 关闭 / 下载 / 主题切换各自命名。

## 4. 关键风险与踩过的坑

1. **stock `TabControl` + WebView2 = 死路**：内容回收会 unparent 控件。必须用 overlay `ItemsControl` + 单 `ContentPresenter` 绑 `ActiveTab.WebView.Control`。这是承重墙。
2. **`OnClosing` 同步 vs 异步拆卸**：`FluentWindow.OnClosing` 是同步的。必须 cancel-then-await + `Task.Yield()`，否则 fire-and-forget 遗留进程，或二次 Close 抛 "window is closing"。
3. **挂起标签拆卸顺序**：已挂起的控制必须先 `Resume()` 再 `Dispose()`，否则可能卡死；每步 try/catch。
4. **`BrowserProcessExited` 是全局事件**：触发后所有标签 Core 已死，走重建路径。
5. **下载 byte 计数类型不稳**：用 boxing + `IConvertible` 转换绕开 `long`/`ulong?` 投影不一致 + WPF 临时工程误报。
6. **ProgressBar.Value 默认 TwoWay**：绑到只读属性会抛 `XamlParseException` 崩溃，必须 `Mode=OneWay`。
7. **Downloads 目录解析**：`%USERPROFILE%\Downloads` 是启发式、用户可能重定位；必须 `SHGetKnownFolderPath(FOLDERID_Downloads)`。
8. **下载 `ResultFilePath` 一次性**：只能在 `DownloadStarting` handler 返回前设（#2890）。
9. **下载 + SmartScreen**：订阅 `DownloadStarting` 同时禁用 SmartScreen 会触发下载被拦（#4612）——只用 `e.Handled`，别动 SmartScreen。
10. **WPF 临时工程（`_wpftmp.csproj`）缓存**：偶发把类型解析成 net462 投影（如把 `ulong?` 误报成 `long`）。遇到类型相关怪报错先 `dotnet clean` 再编。
11. **WPF-UI `Button.Icon` 不能用 Style setter**：必须 inline。

## 6. 已知限制与后续增强

- **页面内焦点快捷键（Ctrl+T/W/L、Ctrl+Tab）未转发**：WPF 的 WebView2 控件**不暴露** `AcceleratorKeyPressed`（它在内部 controller 上）。当前这些快捷键仅在 chrome 有焦点时可用（`Window.InputBindings`），按钮始终可用。完整实现需装**全局 keyboard hook**（`SetWindowsHookEx`，仅本窗口前台时生效）。
- **`MemoryUsageTargetLevel=Low` 未接入**：窗口最小化时主动让浏览器压低内存，比单靠 suspend 更激进，可作为 M5 的补充拿到更明显内存收益。
- **下载面板关闭交互**：当前 `Popup` 用 `StaysOpen` 关闭方式，点击外部不自动关闭（避免与切换按钮的双触发竞态）；可改为失焦关闭。
- **favicon 缓存为内存 `Dictionary`**：进程重启后重取；如需持久化可落盘。
- **多窗口 / 隐身 / 历史 / 收藏**：均未实现（与"精简功能"定位一致）。
